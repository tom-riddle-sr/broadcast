# Go 的鎖機制介紹

在 Go 打造並發程式時，當多個 goroutine 同時存取或修改同一批資料，就會發生「競態條件」(race condition)。鎖就是用來保護資料安全的工具。

---

## 1. 為什麼需要鎖？

假設兩個 goroutine 同時修改一個整數：

```go
var count = 0

// goroutine A
count = count + 1

// goroutine B（同時執行）
count = count + 1
```

執行流程可能是：
1. A 讀 count（0）
2. B 讀 count（0，還沒被 A 存回去）
3. A 存 count（1）
4. B 存 count（1）

預期是 2，實際卻是 1。這就是競態條件。

**鎖的作用**：確保同一時刻只有一個 goroutine 能進入「臨界區」(critical section)。

---

## 2. 最常用：`sync.Mutex`

### 基本用法

```go
import "sync"

var (
    mu    sync.Mutex
    count int
)

// 修改時加鎖
mu.Lock()
count = count + 1
mu.Unlock()

// 讀取時也要加鎖（如果有寫入並發）
mu.Lock()
val := count
mu.Unlock()
```

### 推薦做法：`defer` 避免忘記解鎖

```go
mu.Lock()
defer mu.Unlock()

count = count + 1
// 就算中間發生 panic，defer 也會執行 mu.Unlock()
```

---

## 3. `sync.RWMutex`：讀寫鎖

當大量「讀」、少量「寫」時，普通 `Mutex` 效率不好（讀也會互相等待）。
`RWMutex` 允許「多個讀能同時進行」，但「寫獨佔」。

### 用法

```go
import "sync"

var (
    mu   sync.RWMutex
    data map[string]string
)

// 讀操作：可以多個 goroutine 同時進行
mu.RLock()
val := data["key"]
mu.RUnlock()

// 寫操作：獨佔
mu.Lock()
data["key"] = "value"
mu.Unlock()
```

### `RWMutex` 的三大規則

- `RLock()` / `RUnlock()`：读锁，多個讀可同時進行
- `Lock()` / `Unlock()`：寫鎖，獨佔（會等待所有讀完成）
- 同一 goroutine 不能嵌套加鎖（例如 Lock 後再 Lock = deadlock）

## 4. 在你的 `Hub` 結構中的應用

你的程式碼很好地示範了鎖的使用：

```go
type Hub struct {
	Clients map[userID]*Client  // 多 goroutine 會修改這個 map
	mu      sync.RWMutex        // 保護 Clients 的讀寫
}

// 添加用戶：寫操作
func (h *Hub) AddClient(client *Client) {
	h.mu.Lock()
	defer h.mu.Unlock()
	h.Clients[userID(client.ID)] = client
}

// 移除用戶：寫操作
func (h *Hub) RemoveClient(client *Client) {
	h.mu.Lock()
	defer h.mu.Unlock()
	delete(h.Clients, userID(client.ID))
	client.Cancel()
}

// 讀操作示例（你可能還需要）
func (h *Hub) GetClient(id userID) *Client {
	h.mu.RLock()
	defer h.mu.RUnlock()
	return h.Clients[id]
}

// 廣播給所有用戶
func (h *Hub) Broadcast(msg string) {
	h.mu.RLock()
	defer h.mu.RUnlock()

	for _, client := range h.Clients {
		select {
		case client.MsgChan <- msg:
			// 成功送出
		case <-client.Ctx.Done():
			// 用戶已斷線
		}
	}
}
```

## 5. 常見的鎖相關問題

### `Deadlock`：死鎖

同一 goroutine 嵌套加鎖會卡住：

```go
// 錯誤做法
mu.Lock()
mu.Lock()  // 永遠等不到，因為自己拿著鎖
mu.Unlock()
mu.Unlock()
```

### `Starvation`：飢荒

如果某個 goroutine 一直搶不到鎖，會被餓死。
通常改用 `RWMutex` 或考慮用 channel 替代鎖。

---

## 6. 鎖 vs Channel 的選擇

| 場景 | 用 Mutex | 用 Channel |
|------|---------|-----------|
| 保護共享變數（map、計數器） | ✓ | - |
| 多 goroutine 傳遞訊息、同步 | - | ✓ |
| 需要多讀少寫 | RWMutex | - |
| 複雜的同步邏輯 | - | ✓ |

你的廣播應用其實可以同時用：
- `RWMutex` 保護 client map
- `channel` 傳遞廣播訊息

---

## 7. 實務規範

- **鎖的範圍要小**：不要把整個函式都包，只鎖改變共享資料的部分。
- **避免在鎖內呼叫可能 block 的操作**（例如 I/O、等待 channel）。
- **用 `defer` 解鎖**：避免 panic 時忘記解鎖。
- **考慮用原子操作**：對於簡單整數操作，`sync/atomic` 更快。
- **測試時加 `-race` flag**：`go test -race` 會偵測競態條件。

---

## 8. 一句話總結

**`Mutex` 和 `RWMutex` 是保護共享資料的基礎工具；選對鎖型和粒度，才能既安全又高效。**
